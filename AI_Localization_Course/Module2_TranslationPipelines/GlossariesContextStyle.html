Glossaries, Context, and Style Control

Module Objectives

By the end of this lesson, you will be able to:

  • Create and maintain effective translation glossaries
  • Provide appropriate context to improve translation quality
  • Control translation style and tone through prompt engineering
  • Implement few-shot learning for consistent translations

The Importance of Terminology Management

Inconsistent terminology is one of the most common and damaging localization problems. When "dashboard" is translated as "tableau de bord" in one place and "panneau de contrôle" in another, users become confused. When technical terms are translated incorrectly, documentation becomes unusable. Effective terminology management through glossaries is essential for professional localization.

Traditional machine translation systems struggle with glossaries—they may ignore them entirely or apply them inconsistently. LLMs, however, excel at following glossary instructions when properly prompted, making them ideal for terminology-sensitive translation.

Building Effective Glossaries

A translation glossary is a structured list of terms and their approved translations. Effective glossaries balance comprehensiveness with usability.

Glossary Structure and Format

Glossaries can be structured in various formats, but should always include:

  • Source Term: The term in the source language
  • Target Translation: The approved translation in the target language
  • Part of Speech: Noun, verb, adjective (helps with inflection)
  • Context/Usage Notes: When and how to use this translation
  • Do Not Translate Flag: Terms that should never be translated (brand names, product names)

Example Glossary Entry:

Term: "Dashboard" Spanish: "Panel de control" Part of Speech: Noun Context: Main application interface showing key metrics Do Not Translate: No Notes: Use consistently across all UI elements. Do not use "tablero" or "cuadro de mandos"

What to Include in Glossaries

Not every word needs a glossary entry. Focus on:

  • Product-Specific Terms: Features, components, and concepts unique to your product
  • Technical Terminology: Industry-specific terms with precise meanings
  • Brand Elements: Company name, product names, taglines (often untranslated)
  • UI Elements: Buttons, menus, and interface components that appear frequently
  • Ambiguous Terms: Words with multiple possible translations where consistency is critical
  • Problematic Terms: Words that LLMs frequently mistranslate

Glossary Size and Management

Glossary size involves trade-offs:

Glossary Size Advantages Disadvantages
Small (50-100 terms) Easy to maintain, fits in prompts, high compliance May miss important terms
Medium (100-500 terms) Comprehensive coverage, manageable May exceed prompt token limits
Large (500+ terms) Complete terminology control Cannot fit in prompts, requires dynamic selection

Best Practice: Maintain a comprehensive master glossary but dynamically select relevant terms for each translation request based on content analysis.

Dynamic Glossary Selection

For large glossaries, implement intelligent term selection:

  1. Scan source text for terms that appear in the master glossary
  2. Extract matching terms and their translations
  3. Include in prompt only the relevant subset
  4. Prioritize by frequency and importance if still too many

This approach ensures glossary compliance while staying within token limits.

Context Provision Strategies

Context is the information surrounding the text being translated. LLMs leverage context to make better translation decisions, but context must be provided strategically.

Types of Context

1. Linguistic Context

The surrounding text that helps disambiguate meaning:

Example: The word "bank" could mean financial institution or river edge. Linguistic context clarifies: "I deposited money at the bank" vs "We sat on the river bank."

Implementation: Include 1-2 paragraphs before and after the segment being translated.

2. Situational Context

Information about where and how the text will be used:

Example: "Submit" button on a payment form vs "Submit" button on a support ticket form may require different translations in languages with multiple formality levels.

Implementation: Provide metadata like "UI element: button", "Location: checkout page", "Action: processes payment".

3. Audience Context

Information about who will read the translation:

Example: Technical documentation for developers requires different terminology and tone than marketing materials for consumers.

Implementation: Specify "Target audience: software developers" or "Target audience: non-technical consumers".

4. Cultural Context

Information about cultural norms and expectations in the target market:

Example: Humor, idioms, and references that work in one culture may not translate well to another.

Implementation: Note cultural considerations like "Target market: Japan - prefer indirect communication and high formality".

Context Window Management

LLMs have finite context windows (8K-128K tokens). Effective context management maximizes quality within these limits:

Priority-Based Context Allocation

  1. Highest Priority: The text to be translated (required)
  2. High Priority: Glossary terms relevant to this content
  3. Medium Priority: Immediate linguistic context (surrounding paragraphs)
  4. Lower Priority: Broader document context
  5. Lowest Priority: Related documents or historical translations

Allocate tokens based on priority, ensuring critical information is always included.

Context Formatting

How context is presented affects LLM comprehension:

Structured Context Example:

Content Type: User interface error message Location: Payment processing page Trigger: Credit card declined Audience: E-commerce customers Tone Required: Apologetic but clear, not alarming Previous Context: User clicked "Complete Purchase" button Next Step: User will see options to retry or use different payment method Translate: We're sorry, but we couldn't process your payment. Please check your card details and try again.

This structured approach ensures the LLM understands all relevant factors.

Style Control Through Prompt Engineering

One of LLMs' greatest advantages over traditional MT is the ability to control translation style through natural language instructions.

Defining Translation Style

Style encompasses multiple dimensions:

  • Formality: Casual, neutral, formal, very formal
  • Tone: Friendly, professional, authoritative, playful, serious
  • Voice: Active vs passive, direct vs indirect
  • Conciseness: Verbose, standard, concise, minimal
  • Technical Level: Simplified, standard, technical, expert

Style Specification Techniques

1. Direct Style Instructions

Explicitly state the desired style:

"Translate with a friendly, conversational tone appropriate for consumer-facing mobile app. Use simple language that a non-technical user can easily understand. Prefer active voice and direct statements."

2. Audience-Based Style

Define style through audience description:

"Translate for software developers who are familiar with cloud computing concepts. Use technical terminology accurately. Maintain a professional, informative tone."

3. Comparative Style

Reference known brands or publications:

"Translate in the style of Apple's product documentation: clear, concise, elegant, and user-focused."

4. Example-Based Style (Few-Shot Learning)

Provide examples demonstrating the desired style:

Example 1: English: "Click the button to continue" Spanish: "Haz clic en el botón para continuar" Example 2: English: "Your changes have been saved" Spanish: "Tus cambios se han guardado" Now translate: "Select a file to upload"

The LLM learns from examples to match the style (informal "tú" form, concise phrasing).

Brand Voice Consistency

Maintaining consistent brand voice across languages requires explicit guidelines:

Brand Voice Definition Example:

Brand: TechFlow (B2B SaaS productivity tool) Voice Characteristics: - Professional but approachable - Confident without being arrogant - Helpful and supportive - Clear and jargon-free - Empowering (focus on what users can achieve) Avoid: - Overly casual language or slang - Condescending or patronizing tone - Alarmist or negative framing - Passive voice when active is clearer

Few-Shot Learning for Translation

Few-shot learning involves providing 2-5 example translations that demonstrate the desired output. This is particularly effective for:

  • Establishing Style: Examples show tone, formality, and voice
  • Demonstrating Format: Examples show how to handle special formatting
  • Clarifying Ambiguity: Examples resolve unclear instructions
  • Showing Edge Cases: Examples demonstrate how to handle tricky situations

Effective Few-Shot Examples

Good examples are:

  • Representative: Similar to the content being translated
  • Diverse: Cover different scenarios and edge cases
  • Clear: Unambiguously demonstrate the desired approach
  • Concise: Don't waste tokens on overly long examples

Few-Shot Example Set:

Example 1 - Simple UI text: EN: "Save changes" FR: "Enregistrer les modifications" Example 2 - UI text with placeholder: EN: "Welcome back, {{name}}" FR: "Bon retour, {{name}}" Example 3 - Error message: EN: "File not found. Please check the path." FR: "Fichier introuvable. Veuillez vérifier le chemin." Now translate: "Upload complete. {{count}} files processed."

The LLM learns to: preserve placeholders, use appropriate formality, maintain consistent terminology.

Advanced Techniques

Conditional Translation Rules

Specify different approaches for different content types:

"For UI buttons and labels: Use imperative form, be concise For help text: Use complete sentences, be explanatory For error messages: Be clear about the problem and solution, avoid blame"

Terminology Prioritization

When glossary terms conflict with natural phrasing:

"Prioritize glossary terms for product features and technical concepts. For common words, prioritize natural phrasing over literal glossary matches."

Localization vs Transcreation

Specify when to adapt vs translate literally:

"For marketing slogans and taglines: Adapt creatively to maintain impact and meaning, even if not literal translation. For legal disclaimers: Translate literally and accurately, maintaining all specific terms."

Quality Assurance for Glossary Compliance

After translation, verify glossary adherence:

Automated Glossary Checking

  1. Identify glossary terms in source text
  2. Extract corresponding segments from translation
  3. Verify approved translation appears in target text
  4. Flag violations for review or automatic correction

Handling Glossary Violations

When glossary terms are mistranslated:

  • Automatic Correction: Replace with correct term (for simple substitutions)
  • Retry with Emphasis: Retry translation with stronger glossary emphasis
  • Human Review: Flag for translator review if context makes automatic correction risky

Key Takeaways

  • Glossaries ensure terminology consistency and are critical for professional localization
  • Dynamic glossary selection allows large glossaries while staying within token limits
  • Context provision significantly improves translation quality but must be managed strategically
  • Style control through prompt engineering enables brand voice consistency across languages
  • Few-shot learning is highly effective for establishing translation patterns
  • Automated glossary compliance checking catches terminology errors early
  • Different content types may require different translation approaches
