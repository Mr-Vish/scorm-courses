Module 1: Foundations of Amazon Q Apps

Card Types and Application Design Patterns

Understanding Card Types

The card-based architecture of Amazon Q Apps provides a library of pre-built card types, each serving a specific function within an application workflow. Understanding these card types and their appropriate use cases is fundamental to effective Q Apps development.

Input Cards

Input cards collect information from users when they run the application. These cards define the parameters that make each application execution unique.

Text Input Card

Purpose: Collects free-form text from users

Configuration Options:

  • Field label and description
  • Placeholder text to guide users
  • Required vs. optional designation
  • Character limits (if applicable)

Use Cases:

  • Collecting customer names, project titles, or topic descriptions
  • Gathering specific questions or search queries
  • Accepting free-form requirements or specifications

Example: An RFP response generator might use a text input card to collect the customer name and project description.

File Upload Card

Purpose: Allows users to upload documents for processing

Configuration Options:

  • Accepted file types (PDF, Word, Excel, etc.)
  • File size limits
  • Single or multiple file upload
  • Instructions for what to upload

Use Cases:

  • Contract analysis applications requiring document upload
  • Resume screening tools
  • Compliance checking applications

Example: A contract risk analyzer would use a file upload card to accept contract documents in PDF format.

Dropdown Selection Card

Purpose: Presents users with predefined options to choose from

Configuration Options:

  • List of available options
  • Default selection
  • Single or multiple selection mode
  • Option descriptions or tooltips

Use Cases:

  • Selecting product lines, departments, or categories
  • Choosing report formats or output types
  • Picking from predefined templates or scenarios

Example: A sales proposal generator might use a dropdown to select the product category (Software, Hardware, Services).

Date Picker Card

Purpose: Collects date or date range inputs

Configuration Options:

  • Single date or date range
  • Date format preferences
  • Minimum and maximum date constraints
  • Default date values

Use Cases:

  • Report generation for specific time periods
  • Event planning applications
  • Historical data analysis tools

Processing Cards

Processing cards perform the core logic of the application by querying data, transforming information, or calling external services.

Q Business Query Card

Purpose: Sends queries to Amazon Q Business to retrieve information from connected data sources

Configuration Options:

  • Query prompt template with variable placeholders
  • Data source scope (all sources or specific sources)
  • Response length preferences
  • Context instructions for query interpretation

How It Works: The card takes inputs from previous cards, inserts them into the query template, sends the query to Q Business, and passes the response to subsequent cards.

Use Cases:

  • Searching knowledge bases for relevant information
  • Retrieving product specifications or pricing data
  • Finding policy documents or compliance requirements

Example: A competitive analysis app might query: "Based on the uploaded RFP document, what are our product capabilities that address the customer's requirements for [requirement category]?"

Text Generation Card

Purpose: Generates formatted text based on inputs and instructions

Configuration Options:

  • Generation prompt with instructions and variable placeholders
  • Output format specifications (bullet points, paragraphs, tables)
  • Tone and style guidelines
  • Length constraints

Use Cases:

  • Creating email drafts or communication templates
  • Generating executive summaries
  • Producing formatted reports or documentation

Example: An email response generator might use: "Write a professional email response to a customer inquiry about [topic], incorporating these key points: [retrieved information]."

Plugin Card

Purpose: Calls external APIs or Q Business plugins to perform specialized functions

Configuration Options:

  • Plugin selection from available integrations
  • Parameter mapping from previous cards
  • Error handling preferences
  • Response parsing instructions

Use Cases:

  • Retrieving real-time data from external systems
  • Performing calculations or data transformations
  • Triggering actions in connected applications

Example: A sales forecasting app might call a CRM plugin to retrieve current pipeline data.

Output Cards

Output cards display results to users in formatted, user-friendly presentations.

Text Display Card

Purpose: Shows generated or retrieved text to users

Configuration Options:

  • Formatting options (headings, bold, italics)
  • Section labels and descriptions
  • Copy-to-clipboard functionality
  • Download options

Use Cases:

  • Displaying analysis results
  • Showing generated content
  • Presenting recommendations or insights

Formatted Output Card

Purpose: Displays structured data in tables, lists, or other formatted layouts

Configuration Options:

  • Layout templates (table, list, grid)
  • Column or field definitions
  • Sorting and filtering options
  • Export capabilities

Use Cases:

  • Presenting comparison data
  • Showing search results
  • Displaying structured analysis outputs

Application Design Patterns

Effective Q Apps typically follow established design patterns that combine cards in proven sequences.

Pattern 1: Simple Query and Display

Structure: Input → Query → Output

Description: User provides a query parameter, the app searches data sources, and results are displayed.

Example Application: Policy Lookup Tool

  • Text Input: "What policy topic are you looking for?"
  • Q Business Query: "Find company policies related to [topic]"
  • Text Display: Show retrieved policy information

Pattern 2: Document Analysis

Structure: File Upload → Query (with document context) → Output

Description: User uploads a document, the app analyzes it based on specific criteria, and presents findings.

Example Application: Contract Risk Analyzer

  • File Upload: "Upload contract document"
  • Q Business Query: "Analyze the uploaded contract for potential risks, focusing on liability clauses, termination conditions, and payment terms"
  • Formatted Output: Display risks in categorized table format

Pattern 3: Multi-Step Information Gathering

Structure: Input 1 → Query 1 → Input 2 → Query 2 → Output

Description: The app gathers information in stages, using results from early queries to inform later queries.

Example Application: Competitive Proposal Builder

  • File Upload: "Upload customer RFP"
  • Q Business Query 1: "Extract key requirements from the RFP"
  • Dropdown: "Select your product line"
  • Q Business Query 2: "Match [product line] capabilities to [extracted requirements]"
  • Text Generation: "Create a proposal response section addressing each requirement"
  • Text Display: Show generated proposal content

Pattern 4: Parameterized Report Generation

Structure: Multiple Inputs → Query → Text Generation → Formatted Output

Description: User provides multiple parameters, the app retrieves relevant data, generates narrative content, and presents a formatted report.

Example Application: Executive Briefing Generator

  • Dropdown: "Select department"
  • Date Range: "Select reporting period"
  • Q Business Query: "Retrieve key metrics and initiatives for [department] during [date range]"
  • Text Generation: "Create an executive summary highlighting achievements, challenges, and upcoming priorities"
  • Formatted Output: Display as structured briefing document

Pattern 5: Comparison and Recommendation

Structure: Inputs → Multiple Parallel Queries → Text Generation (synthesis) → Output

Description: The app gathers information from multiple sources, compares options, and provides recommendations.

Example Application: Vendor Selection Assistant

  • Text Input: "Describe your requirements"
  • Q Business Query 1: "Find approved vendors in category [category]"
  • Q Business Query 2: "Retrieve past performance data for these vendors"
  • Text Generation: "Compare vendors based on requirements and recommend top 3 options with justification"
  • Formatted Output: Display comparison table and recommendations

Design Principles for Effective Q Apps

Principle 1: Start Simple

Begin with the minimum viable application that solves the core problem. Add complexity only when necessary.

Principle 2: Clear User Guidance

Every input card should have clear labels and instructions. Users should never wonder what to enter.

Principle 3: Logical Flow

Cards should be arranged in a sequence that mirrors how users naturally think about the task.

Principle 4: Appropriate Granularity

Break complex queries into multiple cards rather than trying to do everything in one query.

Principle 5: Meaningful Outputs

Output formatting should match how users need to consume the information (copy-paste, download, visual review).

Key Takeaways

  • Input cards (text, file upload, dropdown, date) collect user parameters
  • Processing cards (Q Business query, text generation, plugin) perform application logic
  • Output cards (text display, formatted output) present results to users
  • Common design patterns include simple query, document analysis, multi-step gathering, report generation, and comparison
  • Effective Q Apps follow principles of simplicity, clear guidance, logical flow, appropriate granularity, and meaningful outputs
  • Understanding card types and patterns enables rapid application design and development
