MCP Protocol Primitives in Detail

Deep Dive into MCP Primitives

This page explores the three fundamental primitives of MCP - Tools, Resources, and Prompts - in comprehensive detail. Understanding how to design and implement these primitives effectively is essential for building production-quality MCP servers.

Tools: Executable Actions

Tools represent actions that AI can perform. They are the primary mechanism for enabling AI to interact with external systems and modify state.

Tool Design Principles

Effective tool design follows these principles:

  • Single Responsibility: Each tool should do one thing well
  • Clear Naming: Tool names should be descriptive and unambiguous
  • Comprehensive Description: Descriptions help AI understand when to use the tool
  • Validated Inputs: Define strict input schemas to prevent errors
  • Structured Outputs: Return consistent, parseable results
  • Error Handling: Provide clear error messages for troubleshooting

Tool Schema Structure

Tools are defined with the following components:

Component Purpose Example
name Unique identifier for the tool create_user, send_email, query_db
description Explains what the tool does and when to use it Creates a new user account with email validation
inputSchema JSON Schema defining required and optional parameters email (required string), name (optional string)

Tool Categories

Tools typically fall into these categories:

  • Query Tools: Retrieve data without side effects (search, list, get)
  • Mutation Tools: Create, update, or delete data (create, update, delete)
  • Action Tools: Trigger operations (send, execute, deploy)
  • Analysis Tools: Process and analyze data (analyze, summarize, validate)

Tool Best Practices

  • Use verb-noun naming convention (get_user, create_ticket, send_email)
  • Include examples in descriptions to guide AI usage
  • Implement idempotency for mutation tools when possible
  • Return rich context in results to help AI understand outcomes
  • Log tool invocations for audit and debugging purposes

Resources: Contextual Data

Resources provide AI with access to data that informs its responses. Unlike tools, resources are read-only and represent information rather than actions.

Resource Design Principles

  • URI-Based Identification: Each resource has a unique URI
  • MIME Type Declaration: Specify content type for proper handling
  • Discoverability: Resources can be listed and searched
  • Versioning: Consider version information for changing resources
  • Caching: Indicate if resources can be cached

Resource URI Schemes

Common URI schemes for MCP resources:

Scheme Purpose Example
file:// Local file system file:///home/user/docs/readme.md
db:// Database records db://users/schema
config:// Configuration data config://app/settings
git:// Git repository content git://repo/main/README.md

Resource Content Types

Resources support various MIME types:

  • text/plain: Plain text documents
  • text/markdown: Markdown formatted text
  • application/json: Structured JSON data
  • application/xml: XML documents
  • text/html: HTML content

Resource Best Practices

  • Use consistent URI naming conventions across your server
  • Provide resource templates for dynamic content
  • Implement resource subscriptions for real-time updates when appropriate
  • Include metadata (last modified, size, version) in resource responses
  • Consider access control for sensitive resources

Prompts: Reusable Workflows

Prompts are templates that encapsulate common AI workflows. They provide consistency and efficiency for repeated tasks.

Prompt Design Principles

  • Task-Oriented: Each prompt represents a specific workflow
  • Parameterized: Accept arguments to customize behavior
  • Composable: Can reference resources and suggest tool usage
  • Documented: Clear description of purpose and expected inputs

Prompt Structure

Prompts consist of:

Component Purpose
name Unique identifier for the prompt
description Explains the prompt's purpose
arguments Parameters that customize the prompt
messages The actual prompt content with embedded resources

Prompt Use Cases

  • Code Review: Standardized code review workflow with checklist
  • Bug Analysis: Systematic approach to analyzing bug reports
  • Documentation Generation: Consistent documentation format
  • Data Analysis: Structured data exploration workflow
  • Security Audit: Standardized security assessment process

Prompt Best Practices

  • Include clear instructions and expected output format
  • Reference relevant resources within prompts
  • Provide examples of successful prompt usage
  • Version prompts as workflows evolve
  • Test prompts with various argument combinations

Primitive Interaction Patterns

The three primitives often work together:

  • Resource + Tool: AI reads resource to understand context, then uses tool to take action
  • Prompt + Resource: Prompt references resources to provide comprehensive context
  • Prompt + Tool: Prompt guides AI to use specific tools in a workflow
  • All Three: Complex workflows combine prompts, resources, and tools

Choosing the Right Primitive

Scenario Primitive Rationale
AI needs to perform an action Tool Tools execute operations
AI needs background information Resource Resources provide context
Standardize a workflow Prompt Prompts ensure consistency
Real-time data query Tool Tools fetch current data

Key Takeaways

  • Tools enable AI to perform actions and interact with systems
  • Resources provide contextual data that informs AI responses
  • Prompts standardize workflows and ensure consistency
  • Effective primitive design requires clear naming, validation, and documentation
  • Primitives often work together to enable complex AI capabilities
