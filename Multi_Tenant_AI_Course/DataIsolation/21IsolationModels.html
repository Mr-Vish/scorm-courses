Module 2: Data Isolation and Security Strategies

In a multi-tenant AI system, ensuring that one tenant's data is never accessible or visible to another is the most critical design requirement. This is especially challenging when using large language models, as they can inadvertently "memorize" information from their training data or context windows.

2.1 The Three Core Isolation Models

Borrowing from traditional SaaS architecture, there are three primary models for data isolation in multi-tenant AI:

1. The Silo Model (Full Isolation)

In the Silo model, each tenant has their own dedicated resources. This might include a separate database instance, separate file storage, and even a dedicated instance of the AI model.

  • Pros: Highest level of security and performance isolation; no risk of cross-tenant data leakage; easier to meet strict compliance requirements.
  • Cons: Highest cost; complex to manage at scale; poor resource utilization (many idle instances).

2. The Bridge Model (Logical Isolation)

Tenants share some resources (like the application server and the AI model) but have separate databases or separate schemas within a single database.

  • Pros: Better cost-efficiency than the Silo model; easier to manage updates and deployments.
  • Cons: Risk of "noisy neighbor" issues where one tenant's heavy usage impacts others; requires robust application-level logic to prevent cross-tenant access.

3. The Pool Model (Shared Resources)

All tenants share all resources, including the database and the model. Isolation is enforced purely through software logic (e.g., a tenant_id column in every table).

  • Pros: Most cost-effective; highest resource utilization; simplest to scale and manage.
  • Cons: Highest risk of data leakage due to bugs in the application logic; very difficult to provide performance guarantees for individual tenants.

2.2 Preventing Leakage in LLM Contexts

When using LLMs, data isolation extends to the "context" provided in prompts. If a system uses Retrieval-Augmented Generation (RAG), it must ensure that the retrieval step only searches the current tenant's document store.

  • Tenant-scoped Vector Search: Every vector in the database should be tagged with a tenant_id, and every query must include a filter for that specific ID.
  • Prompt Sanitization: Ensuring that no tenant-specific data is "leaked" into system prompts that are shared across tenants.
  • Ephemeral Context: Clearing the model's history or session state immediately after a request is completed to prevent the next tenant from seeing previous data.

2.3 Encryption and Key Management

Robust security requires that each tenant's data is encrypted with their own unique key (Bring Your Own Key - BYOK).

  • Encryption at Rest: Ensuring all stored data is encrypted using tenant-specific keys.
  • Encryption in Transit: Using TLS for all communication between the tenant, the application, and the AI provider.
  • Secure Key Storage: Using Hardware Security Modules (HSMs) or cloud-based key management services (KMS) to manage and rotate tenant keys.

Choosing the right isolation model is a balance between security, cost, and complexity. For many AI startups, the Pool model with strong logical isolation is the starting point, while enterprise-grade systems often move towards the Bridge or Silo models for their most sensitive clients.
